🔍 Key Findings

What we learned from 14 test rounds

❌ ERC20Transfer has NO data field▶
Wallet-core ERC20Transfer protobuf has no bytes data. Any data in an EIP-681 URI is stripped during transaction building. This is the root cause of all failures. Source: tx_builder.rs in wallet-core.
✅ Transfer / ContractGeneric HAVE data▶
Native ETH transfer and generic contract interaction protos both carry bytes data. But when these routes trigger, wallet shows ETH amounts — not USDT. The display is wrong even when data passes.
⚠️ Closest result: v5/C ($0.04 gas paid)▶
/drain?addr=RCP&uint256=1M&data=... — transaction executed on mainnet, gas paid, but data was stripped → no USDT moved. Proves confirm button works for generic paths but data doesn't survive.
💡 Dapp Browser = Best remaining angle▶
trust://browser?url=... opens dapp browser with Web3 provider. Dapp calls eth_sendTransaction directly — no EIP-681 parser, no data loss. UNTESTED.
🧭 Wallet-Core Routing Logic▶
URI → EIP681Parser
├─ Has amount? → ERC20Transfer (data ignored)
├─ Has /transfer+addr+uint256? → ERC20Transfer (data ignored)
├─ Has /approve+addr+uint256? → ContractGeneric (data MAY pass)
├─ Has value (no amount)? → Transfer (data passes, shows ETH)
└─ Has data only? → ContractGeneric (data passes)
